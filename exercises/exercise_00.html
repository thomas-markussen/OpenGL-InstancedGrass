Exercise 00 - Getting started with OpenGL

Learning objectives

The purpose of this set of exercises is to get started with OpenGL. You will:

  • Setup an OpenGL application from scratch.
  • Create a GLFW window and an OpenGL context.
  • Setup the buffers necessary for rendering with OpenGL (Vertex Buffer Object (VBO), Vertex Array Object (VAO) and Element Buffer Object (EBO)).
  • Draw simple shapes.
  • Develop a basic understanding of the Normalized Device Coordinates (NDC) and of the window coordinates.

Exercise 00.1 - Refactor Vertex Buffer Object (VBO) and Vertex Array Object (VAO)

To continue with the refactor, this time on your own, we will focus on the VBO and VAO that we explained briefly during the lecture.

When moving the code to the corresponding methods in our own classes, try to understand what it does. Don't be worried if it is not completely clear, we will take a closer look at these objects during the next lecture. Also, don't hesitate to ask questions if there is something you don't understand.

Make sure that you compile and run your code at every step.

You will find some hint comments in the code, tagged with the label "(todo)" followed by the number of the exercise.

A short checklist of the things you should refactor:

  1. Locate and include the header for the new classes: VertexBufferObject and VertexArrayObject. Additionally, you need to include another header named VertexAttribute.
  2. Replace the integer handles VAO and VBO with local variables of the corresponding new classes. I recommend using lowercase variable names this time.
  3. Find the constructor and destructor for the new classes. Fill them in by moving the corresponding glGenXXX and glDeleteXXX functions. For VertexBufferObject you also need to look in the base class.
  4. Binding is required to make your object active before performing operations on it. Replace all the glBindXXX with the methods of the new classes. When binding a 0 handle we will use our new Unbind method instead, to reflect the real intention. Don't forget to implement those methods in their .cpp files as well.
  5. Fill in the VBO with the vertices data. In OpenGL this is done using glBufferData. We named this function AllocateData in our library, to remind us that it is allocating memory when we use it.
    Implement both versions of the function (with and without initial data) in the library, and use the one that receives a std::span.
    To create a std::span with the vertices you just need to specify a pointer to the vertices and the number of elements. You can hardcode the count directly (9), or you can compute it dividing the size of the vertices array by the size of one float.
  6. Finally, we need to tell the VAO where to look for the vertex data. This is performed in 2 instructions in OpenGL (setting the attribute pointer and enabling the attribute), but you have to implement both in one single function (SetAttribute) in our library.
    We have grouped some of the parameters required here under a new type named VertexAttribute, that defines the size and the data contained by an attribute. Create a VertexAttribute named "position", with type float and size 3.
    Set that attribute in the VAO, in the location 0, and with offset 0. We will get back to this during next week lecture and exercises.

Exercise 00.3 - Basic quad

The triangle is nice, but now we would like to have a quad (rectangle). To build it, we need at least 2 triangles.

First, modify the last vertex (top) in the vertices array to create a right triangle (90 degrees in one corner). One way to do it is to change the X coordinate to 0.5

Add another 3 vertices to the list. Two of them will be repeated from the first triangle (the side shared by both triangles), and the other one will be opposite to the right angle. If you run the program now, you will still see only 1 triangle, that is normal.

Find the Allocate for the VBO and update the count, from 18 to 12.

Find the glDrawArrays call and change the number of vertices drawn to 6 instead of 3. You should be able to see a quad now!

Things you can try if it is not working:

  • Have you allocated data for it in the VBO? If you are not computing the size automatically, you need to change it to 18 manually.
  • Is the position of the vertices properly set? Try to figure out the position of the vertices, use pen and paper if you need to.
  • Ask the teachers or the TA.

Exercise 00.4 - Adding an Element Buffer Object (EBO)

As you could see in the previous exercise, adding the 3 vertices for each triangle can be redundant. This gets even worse when each vertex contains more information than just the position.

To solve this, we can add each vertex just once and use a list of indices to reference those vertices. This indices are stored in an Element Buffer Object (EBO).

In this section you have to add an EBO to optimize our quad. This follows the second part of the Hello Triangle tutorial, so feel free to go back to it if you get stuck or need more information.

Recommended steps:

  1. Remove the 2 repeated vertices from the array. You should have now 4 vertices, all different from each other.
  2. Find the Allocate for the VBO and update the count. At this point you should already regret not computing it automatically as suggested in the exercise 00.2.
  3. Create an array of unsigned int. The initial contents of the array should be: {0, 1, 2, (first triangle), 2, 0, 3 (second triangle)}
  4. Locate and include the header for the new class: ElementBufferObject.
  5. Create a new local variable "ebo" of type ElementBufferObject, next to the VAO and VBO.
  6. We don't need to add the glGenBuffers and glDeleteBuffers this time, because ElementBufferObject and VertexBufferObject use the same base class. Same for glBindBuffer and others. The refactor is starting to pay off!
  7. With our VAO still bound, bind the EBO. This let the VAO know that it needs to use this EBO for rendering. Important: Do not unbind the EBO until the VAO is unbound! Otherwise the VAO will stop using the EBO.
  8. Right after binding the EBO, we can take this chance to Allocate the indices data into it. You have to use a std::span with the indices pointer and the indices count. The indices count can be computed dividing the size of the indices array by the size of an unsigned int.
  9. After the VAO is unbound, we can also unbind the EBO.
  10. Finally, we need to change the function used to draw the geometry. Instead of glDrawArrays we need to use glDrawElements. The parameters for it are slightly different:
    • The first parameter is the type of primitive (same as before).
    • The second parameter is the number of indices to be drawn (6, same as we had before as third parameter). If you computed automatically the indices count, you could reuse the variable here.
    • The third parameter is the type of the indices. In this case we will use GL_UNSIGNED_INT.
    • The fourth parameter is the offset where to start counting inside the EBO. We start from the beginning of the buffer, so the offset is 0.

If everything went well, you should have the same quad as before!.

Exercise 00.5 - Rotating quad

All the previous exercise allocate the geometry data once and then render it every frame. This is, by far, the most common use case, but it is also possible to modify the vertex data dynamically.

Using the same code we already have for the quad, we are going to update the VBO data every frame to create the illusion of a rotating quad. This is not the best way to do it, but we will see more about that in lectures 2 and 3.

Recommended steps:

  1. Resize the window to a square size. This is because the positions are in NDC (normalized device coordinates) and they are relative to the window size.
  2. Before computing the new values of the vertices, we need to know how to modify the VBO. If we used the Allocate function every frame, it would be a waste of memory. For that, OpenGL has a function named glBufferSubData. Implement this function in our BufferObject class, inside the UpdateData method.
  3. Try manually modifying any vertices and updating the VBO before continuing, to verify that it works. Bind the VBO before using UpdateData.
  4. For any animation, we need time. Create a "float time" variable and increase the value inside the main loop. For now, you can increase a fix amount (0.1, for instance), but ideally it should be using the real elapsed time.
  5. Using the time variable, compute a "float angle" every frame that will be angle = time * rotation speed. The angle units are radians.
  6. Look at the diagram below. The position of each vertex can be computed rotating a vector by an angle. The first vertex is the square angle + 45º, then each of the next are separated another 90º from the previous one.
  7. You can simply use std::sin(angle)*length for the X coordinate, and std::cos(angle)*length for the Y coordinate. Z will remain as 0. If we want to keep the same square size, the length of the vector is sqrt(2)/2.

Did it work? Don't hesitate to ask your teachers if you get stuck with the math. And remember to update the vertices every frame!

Exercise 00.6 - Circle

Using the same math described in 00.5, create a circle instead of a quad (not moving is fine).

You will need to generate procedurally the vertices AND the indices. You can use std::vector instead of an array if you want.

I recommend adding a vertex at the center (0,0,0) and using it to form a triangle with every two consecutive vertices of the border.

Don't forget to update the count in the glDrawElements function, if it is not computed automatically.
