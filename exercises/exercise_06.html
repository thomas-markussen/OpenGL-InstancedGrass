Initial implementation

The project provided contains a single scene with a few models. All the models are using the simple lit shader we discussed during the lecture. Your goal is to create 2 new shaders based on that one, implementing to NPR techniques: cel-shading and cross hatching.

In this document we are working on the teapot model, but feel free to test your shaders on other models (you can try importing new ones as well).

There is a folder named "Shaders" that contains some shader code to help you with the exercise:

  • Sample shader files (SampleLit and SampleUnlit): The shaders we discussed during the lecture, with extra comments.
  • Template shader files (TemplateLit and TemplateUnlit): Basic skeleton of a shader, so you can use it as a base for your own shaders in this and future exercises
  • Helper functions library (ITUCG.glslinc) : Some useful functions with comments.
Exercise 6.0
 Setting up a new shader

For the first part of the exercise we are going to implement a cel-shading shader. But before we get into that, we need to know how to create new shaders.

We will start from the Lit shader we implemented during the lecture. Follow this steps:

  • Create a new shader, copying the provided template for lit shaders (TemplateLit.shader), and rename it to CelShading.
  • Open your new shader file and, in the first line, change the Shader name to "CG2024/CelShading".
  • Create a new material, rename it to CelShading, and select your shader name from the list, instead of Standard.
  • Assign it to any of the objects in the scene (just drag and drop on top of it).
Exercise 6.1
Cel-shading

Cel shading is a type of non-photorealistic rendering that makes 3D objects look like cartoons: https://en.wikipedia.org/wiki/Cel_shading

We are starting with the Blinn-Phong lighting model already implemented. This gives us a nice smooth gradient, but this is not what we are looking for. What we want to do is to get a continuous light intensity value and divide it in several discrete levels.

Try this:

  • We want to compute the shading levels independent from the albedo of the object. To do that, in the lighting equation, we can replace it with a full white value (vec3(1.0f)).
  • The lighting we are computing now is colored, and we said that we only want to have the intensity at this moment. For this, you can use the function named GetColorLuminance in the helper shader library we included, ITUCG.glslinc.
  • The output of the fragment shader will be the intensity multiplied by the albedo and the _LightColor0.
  • Add a new property named _Levels, of type Int and default value 3. This will define how many shades we want in our shader. For more information about adding properties in ShaderLab, check here.
  • Add an int uniform with the same name, next to the other uniforms.
  • Finally we are ready to discretize our intensity. This can be done in many ways, the want we recommend is very simple:
    • Multiply the intensity by the _Levels.
    • Round to an integer value. You can use different roundings, try with floor, round and ceil, and observe the difference. I recommend ceil to avoid hard dark shadows, but use the one you like the most.
    • Divide the result again by the _Levels.
  • Don't forget to repeat the steps for the additive pass!.

You should have a nice looking cel-shading shader now :)
Please take some time to play with it in different light settings and different models.
Exercise 6.2
Outline

I know what you are thinking. That doesn't look like a cartoon, it is missing an outline! We will take care of that next.

There are 2 main approaches when rendering outlines. One would be using post-processing and edge detection. The other one, simpler, requires rendering the objects in a different pass, extruded in the direction of the normal, without color and culling the front faces. We will be using this second technique.

As we just said, we have to create a new pass. You can find more information on shaderlab passes here. We want this pass to execute once during the lighting pass, that means that the lightMode will have to be ForwardBase. Step by step:

  • Duplicate the ForwardBase pass, our main pass, but empty the content of the main() function in both vertex shader and fragment shader. You can also get rid of anything related to vertexToFragment here, we won't interpolate any variables in this pass.
  • Change the name to "OUTLINE", but keep the LightMode as ForwardBase
  • In the fragment shader just output black
  • In the vertex shader we need to do the extrusion, in world space:
    • Transform the vertex position to world space.
    • Transform the normal to world space.
    • Add the normal, multiplied by a small number, 0.01f for instance, to the world position.
    • Output to gl_Position the position, transformed by the view projection matrix. The name of the view projection matrix in Unity is unity_MatrixVP
  • If you try the shader now, it will look all black. We are missing the last trick, culling the front faces. Check how to do it here.

Congratulations! This concludes the cel-shading shader exercise. If you want, you can add a Float property Thickness to replace that 0.01f that we hardcoded there, but it is optional.
Exercise 6.3
Cross hatching

Crosshatching is a technique that utilizes parallel, intersecting lines to shade an illustration: https://en.wikipedia.org/wiki/Hatching.

As with cel-shading, we want to divide the lighting received in discrete levels. But this time, each one will be using a different texture, with different density of black lines over white background.

Since you already know how to set up a new shader, we have done the boring part for you in this exercise. You should modify the existing shader "Hatching.shader", in the folder with the same name.

The shader is slightly based on the previous exercise, with the albedo already split from the lighting. Take some time to get familiar with it before you start doing changes. After that:

  • Add 6 texture 2D material properties to the shader: Hatching0 to Hatching5. Assign to your material the textures provided with the same numbers.
  • Add their corresponding uniforms with the same name. There is no need to add texture transforms for all the textures, only the first one. Remember, texture transforms are vec4 and take the name of the texture with a "_ST" suffix.
  • In the vertex shader of your passes, compute the texture coordinates that will be used for hatching and pass it to the shader. Notice that you can pack it with the albedo texture coordinates. Do this for both passes, ForwardBase and ForwardAdd.
  • There is no need to modify the fragment shader this time, we will work in a function in the common include block: ComputeHatching.
  • As we did in the first exercise, we need to get a scalar intensity from the color luminance.
  • In order to avoid lighting artifacts, clamp your intensity value to be between 0 and 1.
  • Get an intensity level scaling your intensity by the number of levels. No need to expose a levels property, the number of levels is fixed: 7. 6 textures + some blending to black and white.
  • What we want to do now is to take 2 values, depending on the intensity level, and mix them. We will get the to values like this:
    • From 0 to 1, we will use black as the first value and sample Hatching5 texture for the second.
    • From 1 to 2, we will sample Hatching5 and Hatching4 textures.
    • From 2 to 3, we will sample Hatching4 and Hatching3 textures.
    • ...
    • From 5 to 6, we will sample Hatching1 and Hatching0 textures.
    • From 6 to 7, we will sample Hatching0 texture and use white as the second value.
  • Finally we have to mix the values we sampled. The fractional part of the intensity level will be how much we take from each value. That will be our hatching intensity

Still one thing left: Copy the entire outline pass from the previous exercise!

Again, take some time to play with this shader in different models and different lighting.
Exercise 6.4
Cross hatching - Texture array

While the previous solution visually works, the code is a bit difficult to maintain with that many textures. Would it be possible to optimize it using a texture array?

The idea is to avoid conditionals. We still need to mix to samples, but already know which texture we are sampling (the texture array). It is only a matter of finding which slices of the array to use. The slice is passed to the texture function as a third component in the texture coordinates.

  • Add a new material property, with type 2DArray. You can find a sample texture array to use in the same folder: hatchArray
  • Add the corresponding uniforms for this. Remember that the sampler must be sampler2DArray. Since we are going to replace the other textures, you should add a uniform for the texture transform as well.
  • Now, the samples. The first one will get the slice rounding down the level, and the second one rounding up. Remember the rounding functions we saw in the first exercise?.
  • Mix those two samples with the fractional part of level, and that's it!

You should get the same results than before but using only one texture property, which makes it more efficient but also easier to use. Imagine if you wanted to change to a different pattern!
