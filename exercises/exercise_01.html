Exercise 01.1 - Creating a 2D grid of quads

As a base for our terrain, we want to create a 2D grid of squares. Each square is made of 2 triangles, and has a size of 1 x 1. The positions should look like in the image.

The size of the grid (number of squares per side) is given in the constructor and stored in member variables m_gridX and m_gridY.

There are some helper classes (i.e. Vector3) that you can use to store data instead of floats.

Recommended steps:

  1. In TerrainApplication.h, add member variables for a VBO and a VAO. Don't forget to include their required headers.
  2. In Initialize(), create the vertex data:
    • Using 2 nested loops, create a list of positions for the grid. Each quad will need 6 vertices (2 triangles). I recommend using some container, like std::vector
    • Allocate the data for the VBO, with the value of the positions.
    • Set the attributes pointer in the VAO.
    • You can use last week exercise as a reference, but try to remember the steps by yourself. Use the lecture slides if you need to.
  3. In Render(), draw the VAO using glDrawArrays.
  4. Nothing to do for Update() (nothing dynamic) or Cleanup() (VBO and VAO will release the resources in their destructors).

The result should show a white square from the center to the top-right corner of the screen. This is not what we want, but we will investigate it in next section.

Exercise 01.2 - Analizing GPU captures with RenderDoc

Working with 3D graphics, there are many times when the results are not what we expect, but reviewing the code does not reveal any issues (maybe you have been in that situation already). In those cases, the best we can do is take a GPU capture and analize it.

A GPU capture is a list of all the commands executed by the GPU during one frame. You can replay it step by step and find out where the problem is.

During this course we will use RenderDoc, a free, open source, graphics debugger. Unfortunately, it is not supported on Mac. It is not required for the course, but it is a great tool when you get stuck. If you are using macOS for the exercises, you can skip this one or pair with someone that is on Windows/Linux.

After you installed and executed RenderDoc, you have to launch your exercise from there. Find the "Launch Application" tab, and configure find your executable. It will automatically set working directory, which is fine for this week exercise. However, after next week, you should use the folder where your code and data is located as the working directory.

With your exercise running connected to RenderDoc, you can take GPU captures pressing F12. Take one capture and exit your application, but not RenderDoc. If it does not open the capture automatically (if you took more than one, for instance), you will have to open by double clicking on it.

In the left menu ("Event Browser") you can find all the relevant OpenGL commands that your application executed this frame. If you select glDrawArrays, you can inspect the parameters for the function in the "API Inspector" window

Now, to inspect the geometry we just created, we can use the "Mesh Viewer" tab. There you can find the data for every vertex, and a preview of your grid, before and after being processed.

If everything went ok, we can see that the grid looks fine in the "VS In" preview, but not in "VS Out". If you move the camera in "VS Out" you will notice the problem: the grid is too large! Move to next exercise to fix it.

Exercise 01.3 - Framing the grid

As we observed in our GPU capture, the grid is to big to fit in the window. This makes sense, because each quad is already 1x1, and normal device coordinates (NDC) go from -1 to 1.

In this exercise you must modify the code where you set the position of the vertices, using a scale of 1/m_gridX x 1/m_gridY, so the total size of the grid is 1.

For it to be centered, you also need to substract 0.5 to each vertex, both in X and Y axes.

After this, you should be seeing a white square centered in the window, but it is difficult to see that it is a grid. However, it is possible to enable wireframe mode easily! Exercise 01.4 - Vertex attributes: Texture coordinates

We are going to add a new vertex attribute, texture coordinates (Vector2) to the grid. Texture coordinates are useful for mapping textures (as we will see in lecture 4). Each quad should be mapped as shown in the image.

You can follow these steps:

  1. Declare a new list of 2 floats for each vertex (std::vector<Vector2>).
  2. Inside the nested loops where you fill in your position data, do the same for texture coordinates. Observe that positions require a Vector3 and texture coordinates a Vector2.
  3. Change the data allocation for the VBO. We want these two attributes in the same VBO, so we will allocate the size for them, without initial data. You can compute the size for each array multiplying the number of vertices by the size of each attribute.
  4. Once the uninitialized data is allocated in the VBO, you can set the attribute data one by one using UpdateData. You will need to specify an offset: positions will start from the beginning of the buffer (offset 0), and texture coordinates will be right after them (offset is equal to the size of the positions).
  5. Finally, set the attribute pointer for the new attribute on the VAO. This new attribute will use location 1, and you need to specify the same offset that you used before in UpdateData. In both attributes, stride can be 0 (default), which means that it should use the size of the attribute.

We have included a debug mode to visualize the attributes of the grid. Press '1' key to see the texture coordinates, and '0' to go back to white. You can disable wireframe mode to see the colors properly.

Exercise 01.5 - Add an EBO

Before we continue modifying the grid, we have to eliminate redundancy. For every quad, we currently are adding 6 vertices, with 2 of them being a copy of another. This is not efficient, and it is also a potencial source of errors (if we modify a vertex but not the copy, for instance).

Most of the code you should be able to figure it out yourself with the help of the slides and last week exercise.

The number of indices in the EBO will be: 2 * 3 * gridX * gridY. The same as we had for positions before, so we can expect that the code will be similar. However, the new number of vertices is (gridX + 1)*(gridY + 1). This is one more iteration than the previous loops, so bear that in mind.

There is another thing to pay attention to. For texture coordinates we were assigning different values to vertices that shared the same position. This is not possible anymore, because those vertices are shared now with the EBO. The solution to this is to assign sequential coordinates to the texture coordinates, exactly as we were doing for the positions in exercise 01.1 (see image there). In lecture 4 we will learn about how the texture will still be able to map that to the same 0-1 range, taking the fractional part.

Don't forget to replace glDrawArrays with glDrawElements.

The key to an easy solution is to visualize the index number that we assign to each vertex. If you get stuck, don't hesitate to draw the grid manually or, even better, use your new RenderDoc skills.

The result should be identical to the one in the previous exercise.

Exercise 01.6 - From grid to terrain

Now that we have our optimized grid, it is time to go 3D, adding height. Of course, the way we look at the grid now, it is not possible to tell the height, if it had. But we have a secret for you... if you press the TAB key, you will change to a different camera! We will learn all about that in lecture 3. In the meantime, just use that camera that we have preset for you.

To calculate random values for height, we will use an external library, stb_perlin. It is a single header library, which means that you can include that file many times, but in one (and only one) of them you have to define the symbol STB_PERLIN_IMPLEMENTATION before including the header. We will only include it in TerrainApplication.cpp, so we will have to define that in there.

The documentation for the library is also included in the header file. Take some time to look at it, and use the library to generate the Z component of the vertex position, using its X and Y as parameters. I recommend that you use stb_perlin_fbm_noise3.

In addition of the function parameters described in the documentation, you can play with the frequency (multiplying X and Y parameters) and the amplitude (multiplying the result returned).

Feel free to enable wireframe mode again to visualize the grid better.

You can also change the number of subdivisions (m_gridX and m_gridY) in the constructor, to get more details in your terrain.

Exercise 01.7 - Vertex Color

We can improve the look of the terrain by adding color.

Following the same as steps as the texture coordinates, add a new "color" attribute, using a Vector3:

  • Add an std::vector for colors.
  • Fill it in with a constant value for now.
  • Create the VertexAttribute.
  • Update the VBO, adding the new data. The offset is the texture coordinates offset + the size of the texture coordinates data.
  • Set the vertex attribute pointer in the VAO, with index 2, using the same offset.

You can visualize if the new attribute is working properly pressing the key "2" during the execution.

Now we will tint each vertex with different colors, depending on the height (position.z). With a simple if/else sequence comparing the height to a constant value, divide the terrain in categories (water, sand, grass, rock, snow...).

Disable the wireframe mode once again to see the results.

Do you notice some strange glitches? This is because the depth test is disabled by default, so the triangle order determines what gets drawn. It was like that before, but it was not noticeable.

You can enable depth testing with this instruction: glEnable(GL_DEPTH_TEST);. If you are interested, it is possible to see the contents of the depth buffer using RenderDoc.

Exercise 01.8 - Interleaved data

As we discussed during the lecture, the vertex data can be stored in different ways inside the VBO.

Currently we are storing the data for each attribute contiguous in memory. Then, we place each attribute list one after the other.

We are going to change that to an interleaved layout:

  • Create a struct Vertex, that will have all the attributes for a single vertex.
  • Replace the data for the 3 attributes with a single array/std::vector of type Vertex.
  • Fill in the vertex data as before, adding Vertex will all the attributes.
  • Now that all data is in a single data structure, we can Allocate with initial data in one line, instead of using UpdateData.
  • Finally, we still have to set the attribute pointers in the VAO. They will use the same indices and VertexAttributes, but the offset and stride are different. For offsets, each attribute will use the offset of the previous one + the size of the previous attribute. The first attribute, position, has an offset of 0. For the stride, all the attributes have the same stride, the size of the Vertex structure.

If everything went ok, we will have the same terrain as before, but it will be easier to add new attributes.

Exercise 01.9 - Vertex normals

Normals are a common vertex attribute that is closely linked to lighting.

Add a new attribute Vector3 normal to your data. Same steps as the others before. Initialize it to any value when creating the vertex, because at that time we still don't know the actual value.

To compute the normals we need to compare the heights of the neighbouring vertices:

  • Repeat the 2 nested loops, and for each vertex access the vertices to the left, right, up and down.
  • If a vertex is sitting on an edge, use it instead of the not existing neighbor. For instance, if it is at the left, use the vertex itself instead of the left vertex.
  • Obtain the delta in X for that vertex: (right.z - left.z) / (right.x - left.x).
  • Obtain the delta in Y for that vertex: (up.z - down.z) / (up.y - down.y).
  • The normal can be computed creating the Vector3 (deltaX, deltaY, 1) and then normalizing it (divide by its length).

You can visualize the normal data pressing the key "3".
