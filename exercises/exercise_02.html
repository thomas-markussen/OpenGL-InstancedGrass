Exercise 02 - Particle systems

Learning objectives

Particle systems play an important role in video games. Even though during the course we will focus on 3D models, today you get a chance to play with particles. While doing that, you will:

  • Get familiar with shader compilation and linking.
  • Understand the differences between vertex and fragment shader types.
  • Learn to pass data to shaders as uniforms.
  • Connect different shader stages with in and out attributes.
  • Use time to animate the shaders.

Remember that you can find hints in comments in the code, tagged with the label "(todo)" followed by the number of the exercise.

Exercise 02.2 - Random point size

Now that the code is stable you should take some time getting familiar with it. The code has been designed to be very easy for you to add new features.

The first change we will do is to change the size of the point particles. You can easily change it using glPointSize during the Initialize function. Try using different values and observe the result.

That was easy, right? Unfortunately, if we want to have each particle with a different size, we need to change a few other things.

First, we need to enable the GL_PROGRAM_POINT_SIZE feature on the DeviceGL class. You can get the instance of the application with GetDevice(). This will let OpenGL know that we want to decide the size in the vertex shader.

You can test that (and write your first shader code!) if you assign a value to the built-in variable gl_PointSize in the main() function of particles.vert.

Next, we need to add a new attribute to the particle: "size", of type float. The steps are simple and we will repeat them a few times during the exercise:

  1. Add a new member variable to struct Particle.
  2. Add a new VertexAttribute, with matching type and components, to the s_vertexAttributes array. Please notice that you need to manually increment the fixed size of the array, or it won't compile.
  3. Setting the data in the VBO and setting the attribute pointer in the VAO is done automatically this time, in InitializeGeometry. Please make sure you understand the code, which is very similar to what we have done in past exercises.
  4. Change the EmitParticle function to receive the value of the new attribute as an argument, and assign that argument to the particle inside of that function.
  5. In the Update() function, pass the value that you want for the attribute to EmitParticle.
  6. In the vertex shader, add the new attribute as an in variable (with the same type). The name does not have to match the one in C++. Make sure that you add the layout qualifier, with a location that matches the index of the attribute.
  7. Use the attribute as a variable in your shader code.

If you assign your new attribute to the variable gl_PointSize, you should see the effect on the size of the particles.

We can compute a random size for each particle in Update(). We added some helper functions to the class to simplify this. Try different ranges of random values until you are happy with the result!

Exercise 02.3 - Soft circle particles

Althoug we still don't know anything about textures, that doesn't mean that we have to work with the blank square particle. We will try to change it to a soft circle using math.

We already did some changes in the vertex shader, now it is time to change the fragment shader (particles.frag).

When using points, there is a built-in variable that contains the coordinates of this fragment inside the point: gl_PointCoord. The values are between 0 and 1, starting on the bottom-left corner.

Using the value of those point coordinates, we need to obtain a single intensity value that we will store in every component of the output FragColor.

For a circle, we need to convert the range from (0, 1) to (-1, 1). This can be done multiplying by 2 and substracting 1. After that, we can just use 1 minus the length of that vector.

If everything went correctly, you should now see a circle with a gradient from the center to the outside. But there is a problem, when circles overlap, we can still see the black outline of the square.

The solution for this is to use blending. We mentioned it during Lecture 1: Geometry, but we will see it in detail during Lecture 4: Materials and textures.

To use blending, first we have to enable the feature GL_BLEND in the device, as we did before with GL_PROGRAM_POINT_SIZE.

Then, we need to decide how to blend the particles. Since we still don't know how it works, you can just copy this line:

glBlendFunc(GL_SRC_ALPHA, GL_ONE);

This blending is called additive blending, and will add the color of the particle to the background.

Go back to the shader and restore the value of the first 3 components of FragColor to 1. The blending will be controlled by the 4th value, the alpha.

You can try to do other operations with the coordinates and observe the results before moving to the next exercise.

Exercise 02.4 - Particle lifetime

Currently, the particles life forever until the maximum capacity has been reached and they are overwritten by another particle.

By adding the concept of lifetime, not only we can control the duration, but also will be able to animate the particles depending on the age.

We will start by adding 2 new attributes, birth and duration, both of type float. Follow the steps on exercise 02.2 to do that.

For birth, instead of passing the parameter to EmitParticle, we will assign the current time directly.

For duration, you can use a random value between 1 and 2 seconds, for instance.

With birth and duration we delimit the lifetime of the particle, but we are missing the most important part: the current time.

The current time will be the same for all particles and, as such, we will pass it as a uniform.

  • Declare the variable CurrentTime, type float, in the vertex shader, with the uniform qualifier before it.
  • In C++, get the uniform location in the shader program, using its name.
  • With the uniform location, set the uniform EVERY FRAME with the value of the current time. For this, the shader program needs to be in use, so it is better to do it in Render() function.
  • For better performance, get the uniform location only once and store it in a member variable, instead of getting it every frame (it is not going to change, anyway).

Now that we have all the pieces, we can compute the age in the vertex shader by substracting the particle birth time to the current time.

We can't really kill particles, but we can set the point size to 0 in the shader, if the age is greater than the duration.

Do you see the particles disappearing after a while?

Exercise 02.5 - Colors

Ok, let's add some color now!

Again, the first step is to add a new attribute to the particle, so each one can have their own color (do as in exercise 02.2).

For the color you can reuse the Color type that we are already using to clear the background. It is a simple wrapper for 4 float values, in RGBA order.

On the shader side we don't have that class, so you can use vec4 instead.

You can pass a random color to EmitParticle using the helper function.

With those steps now we have the color value in the vertex shader, but the FragColor output is in the fragment shader.

  • To pass the value from vertex to fragment, create a variable Color in the vertex shader, with the qualifier out in front of it (if you already used the name Color for the particle attribute, choose a different name for this one).
  • Still in the vertex shader, inside main(), assign the particle color to the Color variable.
  • In the fragment shader, create the same Color variable, but this time with the in qualifier. Having the same name is enough for the variables to be connected.
  • Assign the Color you just received to the RGB part of the FragColor. For the FragColor alpha, you can multiply the alpha we computed in exercise 02.3 with the Color alpha.

Enjoy your colorful particles!

Exercise 02.6 - Movement

For the last part of the exercise, we are going to add movement to the particle.

Because all the movement has to be computed at once (we can't modify position), it will be something simple: velocity and gravity.

We will give each particle an initial velocity, and for that, you guessed right, we need another attribute.

For velocity we can use a vec2, because all the movement will be in 2D.

Instead of using an initial random velocity, we can inherit the velocity from the mouse. We have the current mouse position, and the mouse position on the previouse frame. Substract one from the other, and divide the result by the delta time to obtain the velocity. I recommend multipling also by a number < 1, otherwise it may feel too fast.

For the gravity, we will use a uniform, as we did with the current time. It is up to you if you want to pass a vec2, or just a float and use it as a vec2 inside the shader (with a value of 0 for the x).
